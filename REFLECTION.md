# Рефлексия

## Почему эта задача

Я выбрал задачу с поиском GitHub-репозиториев, потому что она близка к моей реальной ситуации. Когда начинаешь новую тему, особенно связанную с ИИ, легко утонуть в количестве материалов: статьи, видео, репозитории, фреймворки, старые туториалы, новые библиотеки. При этом новичку сложно понять, где хороший старт, а где проект выглядит популярным, но на практике слишком сложный или уже не поддерживается.

RepoScout решает небольшую, но понятную проблему: помогает быстро получить первичный список репозиториев и объяснение, почему их стоит посмотреть. Мне было важно, чтобы агент не просто красиво отвечал текстом, а использовал реальные данные. Поэтому я выбрал треки A+C: внешний API GitHub и структурированные данные, которые сохраняются в JSON. Это делает поведение агента более проверяемым: можно открыть файл в `data/` и увидеть, откуда взялись рекомендации.

Еще одна причина выбора - ограничение по времени. Я впервые собираю такого агента, поэтому не хотел брать слишком рискованный вариант с обучением модели или сложной оптимизацией. Здесь задача достаточно практичная, но при этом не игрушечная: есть внешний API, есть использование инструментов, есть ранжирование, есть понятные случаи поломки.

## Где агент сломался

Первый проблемный случай - нерелевантная выдача GitHub Search. Если запрос сформулирован слишком широко, например "AI", GitHub возвращает очень популярные репозитории, но они могут быть не лучшими для обучения. Агент частично борется с этим через слова вроде `beginner`, `tutorial`, `examples`, но это не гарантирует идеальный результат.

Второй случай поломки - ограничения API и внешних сервисов. Без GitHub token можно упереться в лимит запросов. LLM-провайдер тоже может вернуть ошибку, если неправильный Base URL, закончились бесплатные запросы или модель временно недоступна. Поэтому я добавил запасной режим: даже если LLM не ответила, агент все равно может вызвать GitHub API и показать ранжированный список по шаблону.

Третий слабый момент - текущая оценка простая. Она учитывает звезды, свежесть обновлений, открытые issues, темы репозитория и слова в описании, но не читает код и не проверяет качество документации глубоко. Поэтому агент честно говорит, что это первичный отбор, а не окончательная экспертиза проекта.

## Что сделал бы дальше

Если бы было больше времени, я бы добавил чтение README для топ-3 репозиториев и отдельный tool, который оценивает понятность документации: есть ли установка, quickstart, примеры, лицензия, активность issues. Это сделало бы рекомендации полезнее именно для новичков.

Еще я бы добавил небольшой web-интерфейс или TUI, где можно менять веса ранжирования: например, кому-то важнее свежесть, кому-то - количество примеров, а кому-то - Python как основной язык. Сейчас эти веса зашиты в коде.

Также я бы сделал кэширование запросов, чтобы не дергать GitHub API повторно, и добавил тесты на ранжирование. Например, можно заранее подготовить несколько репозиториев с разными метриками и проверить, что агент действительно поднимает более учебные и живые проекты выше.

Главное улучшение - более честная проверка пригодности репозитория. Сейчас агент смотрит на внешние признаки, а следующий шаг - анализировать README и, возможно, структуру файлов, чтобы отличать реально учебные проекты от просто популярных.
