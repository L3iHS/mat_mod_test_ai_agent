# Рефлексия

## Почему эта задача

Я выбрал задачу с GitHub-репозиториями, потому что сам часто сталкиваюсь с похожей ситуацией. Когда начинаешь новую тему, сложно понять, какой проект действительно полезен для обучения, а какой просто выглядит популярным. Особенно это заметно в темах про ИИ: материалов много, но быстро выбрать хороший старт тяжело.

RepoScout помогает сделать первичный отбор. Агент принимает обычный запрос, обращается к GitHub API, получает реальные данные о репозиториях, сравнивает варианты и сохраняет отчет. Мне было важно, чтобы это был не просто чат-бот, который отвечает из головы, а агент с инструментами и проверяемыми данными. Поэтому я выбрал трек A+C: внешний API и работа со структурированными данными.

## Где агент сломался

Первый заметный сбой был с маршрутизацией ссылок GitHub. Когда я отправлял ссылку на конкретный репозиторий, агент сначала анализировал весь профиль пользователя. Ответ формально был по GitHub, но не по моему вопросу. Я исправил это отдельным инструментом для конкретного репозитория.

Второй сбой был с LLM-провайдерами. Сначала я пытался подключить AgentRouter, но получил ошибки авторизации. Из-за этого пришлось разбираться, проблема в ключе, URL или коде. В итоге я перешел на GitHub Models и добавил команды `/llm` и `/models`, чтобы проще проверять модель.

Третий слабый момент - глубина анализа. Сейчас агент смотрит на GitHub-метаданные, README и простой чеклист. Это лучше, чем просто текстовый ответ, но это еще не полноценный аудит кода.

## Что сделал бы дальше

Если бы было больше времени, я бы добавил проверку структуры файлов: есть ли `tests/`, `tools/`, `.env.example`, реальные файлы запуска и примеры. Еще полезно было бы сделать evidence-вывод в стиле `path:line`, чтобы агент подтверждал свои выводы конкретными строками из репозитория.

Также я бы улучшил ранжирование. Сейчас score считается по простым признакам: свежесть, звезды, issues, темы и совпадение с запросом. В будущем можно дать пользователю менять веса: например, выбирать, что важнее - Python, документация, свежесть или популярность.

В целом проект уже решает задачу MVP: агент вызывает инструменты, работает с GitHub API, строит таблицы, сохраняет данные и отчеты. Но следующий шаг - сделать проверку репозитория глубже и ближе к настоящему code review.
